1. Project Overview
ClearClaim is a multi-tenant SaaS expense management system. Companies sign up, employees submit expenses, and managers/admins approve them through configurable policy-based approval chains. It features OCR receipt scanning, FX conversion, email notifications, and a Google OAuth flow.

1.2 Architecture Summary
•  Monolithic Next.js application — API routes serve as backend
•  Authentication via HttpOnly JWT cookie with 24-hour expiry
•  Three user roles: ADMIN, MANAGER, EMPLOYEE
•  Multi-tenancy enforced by companyId scoping on every database query
•  PolicyRule engine selects approval chain at submission time and snapshots it on the Expense document
•  OCR pipeline: Gemini 1.5 Flash as primary, Groq as automatic fallback

2.3 Google OAuth Exposes googleId in URL Query Params

2.4 No File Size Limit on OCR Endpoint — DoS / Cost Bomb
[CRITICAL]  No file size limit on OCR endpoint (DoS / cost bomb)
📁 app/api/ocr/route.ts
The OCR route reads the entire uploaded file into memory (file.arrayBuffer()) with no size check. A malicious authenticated user can upload a 500 MB file, exhausting server memory and incurring Gemini/Groq API costs. The file upload endpoint (/api/files/upload) has the same issue — no file size or MIME type whitelist enforcement.
✅ FIX: Add file size and MIME type validation before processing
// app/api/ocr/route.ts — add before arrayBuffer():
const MAX_SIZE = 10 * 1024 * 1024; // 10 MB
if (!file || !file.type.startsWith('image/')) {
  return NextResponse.json({ error: 'Valid image required' }, { status: 400 });
}
if (file.size > MAX_SIZE) {
  return NextResponse.json({ error: 'File too large (max 10 MB)' }, { status: 413 });
}
 
// app/api/files/upload/route.ts — whitelist MIME types:
const ALLOWED_TYPES = ['image/jpeg','image/png','image/webp','application/pdf'];
if (!ALLOWED_TYPES.includes(file.type)) {
  return NextResponse.json({ error: 'File type not allowed' }, { status: 415 });
}
if (file.size > 20 * 1024 * 1024) {
  return NextResponse.json({ error: 'File too large (max 20 MB)' }, { status: 413 });

[HIGH]  In-memory rate limiter resets on every serverless cold start

The buckets Map is process-scoped. On Vercel/serverless, each function instance has its own map, so an attacker hitting different instances bypasses the limit entirely. This affects login (brute-force), signup (spam), and OTP verification. The OTP route has no rate limiting at all.

3.4 OTP Compared With Plain String Equality — Timing Attack
[HIGH]  OTP compared with plain string equality — timing attack possible
📁 app/api/auth/verify-otp/route.ts (line 40)
OTP verification uses user.otp !== otp, which is a non-constant-time comparison. An attacker measuring response timing can potentially determine correct OTP prefix character-by-character. While OTPs are short-lived, this is still bad practice for authentication codes.
✅ FIX: Use crypto.timingSafeEqual for constant-time comparison
import { timingSafeEqual } from 'crypto';
 
function safeCompare(a: string, b: string): boolean {
  if (a.length !== b.length) return false;
  return timingSafeEqual(Buffer.from(a), Buffer.from(b));
}
 
// Replace: if (user.otp !== otp)
if (!safeCompare(user.otp!, otp)) {
  return NextResponse.json({ message: 'Invalid OTP' }, { status: 400 });
}

4. Security Findings
The following 6 security findings range from HIGH to LOW severity. All should be addressed before production.
4.1 ReDoS via Unescaped Regex Search
[HIGH]  Regex search on description is a ReDoS vector
📁 app/api/expenses/route.ts (line 51)
User-supplied search param is used directly in { $regex: search, $options: 'i' }. A malicious input like (a+)+$ causes catastrophic backtracking in MongoDB's regex engine, potentially locking the database thread for seconds.
✅ FIX: Escape regex special characters before querying
function escapeRegex(s) {
  // escape special regex chars
  return s.replace(/special-chars/, c => '\\' + c);
}
// Usage:
if (search) filter.description = { $regex: escapeRegex(search), $options: i };

  session.startTransaction();

  const user = await User.create([{...}], { session });

} catch (err) {
  await session.abortTransaction();
  throw err;
} finally {
  session.endSession();
}

// lib/pagination.ts
export function parsePagination(searchParams: URLSearchParams) {
  const page = Math.max(1, parseInt(searchParams.get('page') ?? '1', 10));
  const limit = Math.min(100, Math.max(1, parseInt(searchParams.get('limit') ?? '50', 10)));
  return { page, limit, skip: (page - 1) * limit };
